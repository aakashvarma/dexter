# Agents

Dexter runs inside OpenCode with three agents. One orchestrator runs the whole pipeline. Two subagents each handle a single focused task. They never talk to each other — the orchestrator is the middleman, passing files and context between them.

The agent prompt files live in `.opencode/agents/`. Their permissions and modes are wired up in `opencode.json`.

---

## The orchestrator — your one contact point

When you open OpenCode and type a prompt, you are talking to the orchestrator. It is the only agent you ever interact with directly.

![Select the orchestrator agent in OpenCode](/assets/images/opencode/select-agent.png)

The orchestrator does not do creative work — it coordinates. It checks what already exists in the run directory before doing anything, so interrupted runs can resume cleanly. It calls subagents when it needs judgment. It runs tool scripts when the work is deterministic. It writes the camera plan for each render cycle. It tracks which placement iteration scored highest. It stops the pipeline at the right moments and waits for your input.

There are two moments where the orchestrator hands control back to you. The first is after the parts list is generated, before any 3D work begins. The second is after the placement loop ends, before the USD is exported.

---

## The analyze agent — reading the photo

The analyze agent's job is to look at your source image and describe the object in a way the rest of the pipeline can use.

It produces `parts.json`. For each moving part it records a name (something specific like `front_door`, not `part_1`), a short description suitable for image generation, which part it attaches to in the hierarchy, and the joint type that connects them. It also adds geometry hints: roughly how large the part is relative to its parent, where it sits, and — for hinged parts — which side the hinge is on, or for sliding parts, which axis they move along.

Because this file is so foundational, the orchestrator always pauses after analyze finishes and asks you to confirm the result.

Here is what a single part entry looks like:

```json
{
  "name": "front_door",
  "description": "The front drop-down door panel with handle on the upper edge.",
  "parent": "cabinet_body",
  "joint_type": "revolute",
  "size_fraction": [1.00, 0.06, 0.60],
  "position_in_parent": "center",
  "hinge_side": "bottom"
}
```

## The critic agent — scoring the result

After every placement round, Blender renders four views of the assembled scene — front, top, left, and isometric. The critic agent looks at those renders alongside the source photo and `world_dims.json`, then writes an honest evaluation.

Its output is `critic.json`: an overall score from 0 to 100 and per-component corrections. For any part that still needs work, the critic provides all applicable fixes together — `suggested_delta`, `suggested_scale_factor`, and `suggested_rotation_delta`. Parts are marked `locked` only when scale, position, and orientation are all correct. `apply_critic.py` applies those corrections deterministically on the next iteration.

| Score | What it means |
|-------|--------------|
| 90–100 | Looks correct; minor cosmetic issues at most |
| 75–89 | Recognisable; one or two clear problems |
| 60–74 | Several issues, but roughly the right shape |
| below 60 | Something is fundamentally wrong |

## Keeping things safe and auditable

Every agent can only write files inside `.intermediate/`. The source tree — your prompt files, schemas, tool scripts — is read-only.

Subagents have deliberately narrow tool access. The analyze and critic agents can only run `validate_json.py`. They cannot browse the web, call arbitrary scripts, or touch files outside the run directory.

No model names or API keys appear in any agent prompt. Subagents use whichever model you have connected to OpenCode.
