# Agents

Dexter runs inside OpenCode with three agents. One orchestrator agent runs the whole pipeline. Two subagents each handle a single focused task. They never talk to each other — the orchestrator agent is the middleman, passing files and context between them.

The agent prompt files live in `.opencode/agents/`. Their permissions and modes are wired up in `opencode.json`.

---

## The orchestrator agent — your one contact point

When you open OpenCode and type a prompt, you are talking to the orchestrator agent. It is the only agent you ever interact with directly.

![Select the orchestrator agent in OpenCode](/assets/images/opencode/select-agent.png)

The orchestrator agent does not do creative work — it coordinates. It checks what already exists in the run directory before doing anything, so interrupted runs can resume cleanly. It calls subagents when it needs judgment. It runs tool scripts when the work is deterministic. It writes the camera plan (`render_views.json`) for each render cycle. It tracks which placement iteration scored highest. It stops the pipeline at the right moments and waits for your input.

There are two moments where the orchestrator agent hands control back to you. The first is after the parts list is generated, before any 3D work begins. The second is after the placement loop ends, before the USD is exported.

---

## The analyze subagent — reading the photo

The analyze subagent's job is to look at your source image and describe the object in a way the rest of the pipeline can use.

It produces `parts.json`. For each moving part it records a name (something specific like `front_door`, not `part_1`), a short description suitable for image generation, which part it attaches to in the hierarchy, and the joint type that connects them. It also adds geometry hints: the root part's real-world dimensions, roughly how large each non-root part is relative to its parent, where it sits, and — for hinged parts — which side the hinge is on, or for sliding parts, which axis they move along.

Because this file is so foundational, the orchestrator agent always pauses after the analyze subagent finishes and asks you to confirm the result.

Here is what a single part entry looks like:

```json
{
  "name": "front_door",
  "description": "The front drop-down door panel with handle on the upper edge.",
  "parent": "cabinet_body",
  "joint_type": "revolute",
  "size_fraction": [1.00, 0.06, 0.60],
  "position_in_parent": "center",
  "hinge_side": "bottom",
  "open_angle_deg": 0
}
```

After writing, the analyze subagent validates `parts.json` against `schemas/parts.schema.json` before finishing.

## The critic subagent — scoring the result

After every placement round, Blender renders four views of the assembled scene — front, top, left, and isometric. The critic subagent looks at those renders alongside the source photo and `assembly.json` (which contains per-part `world_size`, `world_center`, and `rpy_deg` for every link), then writes an honest evaluation.

Its output is `critic.json`: an overall score from 0 to 100 and per-component corrections. For any part that still needs work, the critic subagent provides all applicable fixes together in one issue entry — `corrected_world_size`, `corrected_world_center`, and `suggested_rotation_delta`. Parts are marked `locked` only when scale, position, and orientation are all correct. `apply_critic.py` applies those corrections directly to world-space values on the next iteration.

| Score | What it means |
|-------|--------------|
| 90–100 | All parts correct size and position; no collisions; cosmetic issues only |
| 75–89 | Recognisable; one or two clear problems |
| 60–74 | Several issues, but roughly the right shape |
| below 60 | Something is fundamentally wrong |

Scale errors and position errors carry equal weight — a correctly-positioned part that is 50% the right size is as bad as a correctly-sized part displaced by its own diameter.

After writing, the critic subagent validates `critic.json` against `schemas/critic.schema.json` before finishing.

## Keeping things safe and auditable

Every agent can only write files inside `.intermediate/`. The source tree — your prompt files, schemas, tool scripts — is read-only.

Subagents have deliberately narrow tool access. The analyze subagent and critic subagent can only run `validate_json.py`. They cannot browse the web, call arbitrary scripts, or touch files outside the run directory.

No model names or API keys appear in any agent prompt. Subagents use whichever model you have connected to OpenCode.
