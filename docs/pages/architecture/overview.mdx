# Architecture Overview

Dexter is an **orchestrator-driven agentic pipeline**. One primary OpenCode agent coordinates deterministic tool scripts and four specialized subagents. There is no standalone Python pipeline driver — the orchestrator is the only entrypoint.

## System diagram

![OpenCode agent harness](/assets/images/dexter/opencode_agent_harness.png)

### User and orchestrator

You provide a prompt — for example, *"build the dishwasher from `input_images/dishwasher.png`"*. The **orchestrator** is the only entrypoint (`opencode.json` → `orchestrator` agent). It reads `config.yaml`, probes `.intermediate/<asset>/<NNN>/` for existing artifacts, decides which step to run next, and invokes subagents through OpenCode's Task tool. Subagents never talk to each other; the orchestrator passes context between them.

### Subagents (left to right)

Each subagent writes **one artifact** and exits:

| Subagent | Role in the diagram | Output |
|----------|---------------------|--------|
| **Analyser** | Reads the source image and decomposes the object into moving parts | `parts.json` (Parts IR) |
| **Image Gen** | Does not generate images itself — runs `openai_imagegen.py` per part | `component_images/*.png` |
| **Placement** | Arranges GLBs to match the source image in position and scale | `assembly.json` (Layout IR) per iteration |
| **Critic** | Scores rendered views against the source image | `critic.json` (Critique IR) |

Between subagent steps the orchestrator runs **tool scripts** directly: `build_component_prompts.py`, `fal_image_to_3d.py`, `blender_measure_glbs.py`, `compute_placement_scales.py`, `blender_assemble.py`, `blender_render_views.py`, and finally `blender_export_usd.py`.

**Human gates** (shown as return arrows to the user in the diagram): you review `parts.json` before any 3D generation, and the best placement iteration before USD export (`placement.confirmed`).

**Placement ↔ critic loop** (the **N×** arrow): each round, placement writes `assembly.json`, Blender assembles and renders four views, and the critic returns a score plus per-component fixes. Placement iteration 2+ applies only those corrections. The loop stops when `config.yaml` exit conditions are met; the orchestrator keeps the best-scoring iteration.

### Model runtime

Subagents reason through your logged-in OpenCode model (Codex OAuth). Model names are not hardcoded in the repo — the harness handles LLM calls; Dexter's prompts and permissions live in `.opencode/agents/*.md`.

### Tools

The red **Tools** box is deterministic code the orchestrator runs via bash:

- **Image gen** — `openai_imagegen.py` (`OPENAI_API_KEY`)
- **Mesh gen** — `fal_image_to_3d.py` (`FAL_KEY`)
- **Blender** — measure GLBs, assemble scenes, render critique views, export USD

> **Diagram vs v0 code:** The harness image may show a **Joint Gen** subagent that emits URDF. In the current repo, articulation export is not a separate subagent — after you approve placement, the orchestrator runs `blender_export_usd.py` to produce `robot.usda`.

### Outputs

The pipeline produces two deliverables:

1. **Articulated asset** — per-part GLBs assembled under a parent-child hierarchy, exported as `robot.usda` (+ `textures/`) for Isaac Sim
2. **Blender output** — `iterations/<B>/assembled.blend`, the best visual assembly from the critic loop

All intermediate files accumulate under `.intermediate/<asset>/<NNN>/`. See [Pipeline Run](/getting-started/pipeline-run) for a step-by-step walkthrough with real outputs.

## Repository layout

```
dexter/
├── .opencode/agents/     # Agent prompts (orchestrator + subagents)
├── config.yaml           # Pipeline knobs
├── opencode.json         # Agent definitions and permissions
├── schemas/              # JSON Schema files
├── tool_scripts/         # Deterministic pipeline scripts
├── input_images/         # Source product photos
├── AGENTS.md             # Agent-facing source of truth
└── docs/                 # This documentation site
```

Pipeline output: `.intermediate/<asset>/<NNN>/` (gitignored).

## Downstream consumers

Dexter produces URDF/USD packages consumed by:

- **NVIDIA Isaac Sim** — primary target; PhysX articulation from USD prims
- **Genesis World** — URDF, MJCF, GLB, USD import
- **ROS** — URDF for MoveIt, Gazebo

Dexter does **not** run inside any simulator — it is the step before those tools get useful content.
